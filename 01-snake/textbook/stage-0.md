# Stage 0 — Hello, snake!

> "내 손으로 한 줄을 띄워보는" 첫날. 게임 코드는 아직 한 줄도 없지만, **C라는 언어가 어떻게 컴퓨터를 움직이게 하는가**를 처음으로 만난다.

## 오늘의 목표

- `.c` 파일을 작성하고 `gcc`로 컴파일해서 `.exe`를 만든다.
- 터미널에서 `.exe`를 실행해 `Hello, snake!`를 띄운다.
- "프로그램이 끝나면 창이 사라지는 이유"를 손으로 겪어본다.

---

## 배경 이야기 — `.c`가 `.exe`가 되기까지

우리가 키보드로 두드리는 코드는 사람이 읽는 글자다. CPU는 글자를 모른다. CPU가 아는 건 0과 1로 된 **기계어**뿐이다. `gcc`라는 도구는 이 사이의 통역사다. 정확히는 통역사들의 **공장 라인**이다.

`gcc snake.c -o snake` 한 줄을 치면, 보이지 않게 네 단계가 차례로 일어난다.

1. **전처리(preprocessing)**: `#include <stdio.h>` 같은 `#`으로 시작하는 줄을 처리한다. `<stdio.h>` 파일의 내용을 그 자리에 그대로 펼쳐 넣고, `#define`이 있으면 치환한다. 결과는 여전히 사람이 읽을 수 있는 C 소스 — 단지 헤더가 다 펼쳐진 거대한 버전.
2. **컴파일(compilation)**: C 코드를 **어셈블리어**(CPU에 가까운 사람-읽기-가능한 언어)로 번역.
3. **어셈블(assembly)**: 어셈블리어를 **목적 파일(`.o`)** 의 기계어로 변환. 아직 실행은 못 함 — `printf` 같은 외부 함수의 자리는 비어 있다.
4. **링크(linking)**: 비어 있던 자리에 표준 라이브러리(`libc`)의 `printf` 같은 함수를 갖다 끼워 맞춘다. 그제야 비로소 **실행 파일(`.exe`)** 완성.

`gcc`는 이 네 단계를 한 줄로 묶어 보여줄 뿐이다. 막힐 때 어느 단계에서 실패했는지 알면 디버깅이 빨라진다.

> 💡 옵션을 주면 단계별로 멈출 수 있다: `gcc -E`(전처리만), `-S`(어셈블리까지), `-c`(목적 파일까지). 지금은 아니지만, 컴파일 에러가 안 잡힐 땐 한 번씩 들여다본다.

## 프로세스의 수명 — 콘솔이 깜빡이는 이유

`.exe`를 더블클릭하면 까만 창이 뜨자마자 사라지는 걸 보게 된다. 버그가 아니다. **그게 정상**이다.

OS의 관점에서 `.exe`를 실행하면 **프로세스**(실행 중인 프로그램의 인스턴스)가 새로 태어난다. Windows는 프로세스를 위해 콘솔 창을 열어준다. 프로세스가 `main`의 마지막 `return 0;`에 도달하면 프로세스는 **죽는다**. Windows는 죽은 프로세스를 위해 띄워둔 콘솔을 닫는다. 그래서 창이 깜빡인다.

반대로 **이미 열려 있는 터미널**에서 `./snake`로 실행하면, 콘솔 창의 주인은 터미널 프로세스다. 우리 snake 프로세스는 그 창을 빌려 쓸 뿐. snake가 죽어도 창은 터미널 것이라 안 닫힌다.

> **앞으로 무조건 터미널에서 실행한다.** 더블클릭은 잊는다.

## 새 C 문법

### `#include <stdio.h>`
- 전처리기 지시문. **표준 입출력**(standard I/O) 함수의 선언을 가져온다.
- `printf`, `scanf`, `putchar` 등을 쓰려면 필요.
- `<...>` 꺾쇠는 시스템 헤더(컴파일러가 기본 경로에서 찾음). 내가 만든 헤더는 `"my.h"`처럼 따옴표.

### `int main(void) { ... }`
- 프로그램의 **진입점**(entry point). OS가 `.exe`를 실행하면 여기부터 시작.
- `int` — 반환 타입(정수). OS에 종료 코드를 돌려준다.
- `(void)` — 인자 없음. `int main(int argc, char *argv[])`로 명령줄 인자를 받을 수도 있는데 지금은 안 씀.

### `printf("...");`
- 표준 출력(stdout)에 문자열을 찍는다.
- `\n`은 줄바꿈. `printf("Hi")`와 `printf("Hi\n")`는 화면에 보이는 게 다르다.
- 세미콜론 `;`은 "여기까지가 한 문장"이라는 마침표.

### `return 0;`
- 종료 코드. **0 = 정상**, 그 외 = 에러.
- 셸 스크립트에서 `./snake && echo ok` 같은 분기에 이 값이 쓰인다.

## 코드

`01-snake/snake.c`에 그대로 친다.

```c
#include <stdio.h>

int main(void) {
  printf("Hello, snake!");
  return 0;
}
```

**왜 5줄밖에 안 되나?** 첫날은 환경을 잡는 게 목적이다. `gcc`가 어디 있는지, 터미널에서 컴파일이 통과되는지, 출력이 보이는지를 확인하는 단계. 코드가 길수록 실패 지점이 많아져서 학습이 막힌다.

## 돌려보기

터미널(MSYS2 / Git Bash / Windows Terminal 등 `gcc`가 잡히는 셸)에서:

```sh
gcc 01-snake/snake.c -o 01-snake/snake
./01-snake/snake
```

기대 출력:
```
Hello, snake!
```

> 출력 끝에 줄바꿈이 없어서 셸 프롬프트가 같은 줄에 붙어 보일 수 있다. 정상이다. (Stage 1에서 `\n`을 붙인다.)

## 흔히 빠지는 함정

- **exe를 더블클릭 → 창이 깜빡이고 사라짐.** 위에서 설명한 프로세스 수명 이슈. 반드시 터미널에서 실행.
- **`gcc: command not found`.** `gcc`가 PATH에 없다. MSYS2 설치 후 `mingw64` 환경 터미널을 쓰거나, Windows의 시스템 환경변수에 `C:\msys64\mingw64\bin` 추가.
- **세미콜론 빠뜨림.** C에서 `;`은 문장 종결자. 빼먹으면 다음 줄이 같은 문장으로 이어진다고 판단해 엉뚱한 줄에서 에러가 난다.
