# Stage 2 — 키 한 번 받아 한 칸 이동

> 보드 한가운데에 `O`를 놓고, `w/a/s/d`로 이동시킨다. 처음으로 **사용자 입력**과 **반복 갱신 화면**을 만진다.

## 오늘의 목표

- 보드에 플레이어 `O`를 그린다.
- `w`(위) / `a`(왼쪽) / `s`(아래) / `d`(오른쪽)를 누르고 Enter를 치면 한 칸 이동.
- 벽에 부딪히면 더 이상 안 나간다 (clamp).
- `q`를 누르면 종료.

이 단계에서 만나는 건 정작 게임이 아니라 **터미널이 어떻게 글자를 다루는가**다. 키 한 개를 받는데도 `getchar`가 Enter를 기다리는 것도, 화면을 깨끗이 다시 그리는 것도, 한글이 깨지는 것도 — 다 터미널과 OS의 동작 방식 때문.

---

## 배경 이야기

### 먼저: 프로세스와 스트림이 뭔지부터

Stage 0에서 잠깐 봤다 — `.exe`를 실행하면 **프로세스**가 태어난다. 프로세스 = "실행 중인 프로그램 하나"의 단위. 메모리, CPU 시간, 입출력 통로 같은 자원을 OS로부터 할당받아 살아 있다가 `return 0`에 도달하면 죽는다.

그 입출력 통로의 단위가 **스트림(stream)** 이다. **호스 / 파이프**를 떠올리면 쉽다 — 한 쪽 끝에서 데이터를 부어 넣으면 다른 쪽 끝에서 흘러나옴.

```
[프로세스]  ─── 호스(스트림) ───  [어딘가]
        부어 넣음  →           받음
```

내용물은 그냥 **바이트의 줄**. 한 글자(`'H'`)든 한 줄 메시지(`"Hello"`)든, 결국 호스에 바이트 몇 개를 흘려보내는 일.

### 프로세스가 태어날 때 OS가 자동으로 끼워주는 호스 세 개

매번 호스 연결을 직접 챙기는 건 번거롭다. 그래서 OS는 **어떤 프로세스든 태어나는 순간 세 개의 호스를 미리 연결해서 끼워준다**. 우리는 그냥 그 호스에 쓰거나 읽으면 됨.

| 이름 | 번호(fd) | 방향 | 기본 연결처 |
|---|---|---|---|
| **stdin** (표준 입력) | 0 | 받는 호스 | 키보드 |
| **stdout** (표준 출력) | 1 | 내보내는 호스 | 화면(터미널) |
| **stderr** (표준 에러) | 2 | 내보내는 호스 (에러 전용) | 화면(터미널) |

> `fd`는 file descriptor — OS가 각 호스에 붙여둔 번호표. `printf` 같은 표준 함수를 쓸 땐 우리가 직접 fd를 만질 일 없지만, 셸의 `2>err.log`(stderr만 파일로) 같은 트릭이 이 번호를 이용한다.

`printf("Hello\n");`이 정확히 하는 일은 "stdout 호스에 H, e, l, l, o, \\n 바이트를 흘려보내라". 그 호스 끝이 어디에 연결돼 있는지 **우리 코드는 모른다 — 모를수록 좋다.**

### 그래서 "파일처럼"이라는 게 무슨 뜻인가

이 세 호스는 **파일을 읽고 쓰는 것과 똑같은 인터페이스**(읽기/쓰기/닫기)로 다뤄진다. 파일에 바이트 하나 쓰는 것과 stdout에 바이트 하나 흘려보내는 것은 OS 입장에선 같은 동작.

그래서 셸에서 한 줄만 바꾸면 호스 끝이 바뀐다 — 우리 C 코드는 그대로:

```sh
./snake              # stdout → 터미널 (기본)
./snake > out.txt    # stdout → out.txt 파일 (셸이 호스 끝을 파일로 갈아끼움)
./snake | grep wasd  # stdout → grep의 stdin (파이프)
```

세 경우 모두 우리 `printf` 코드는 한 글자도 안 바뀐다. **호스 끝이 어디인지는 셸의 문제, 우리 프로그램이 신경 쓸 일이 아니다.** 이게 유닉스 철학의 **"모든 것은 파일이다"** — 키보드, 화면, 진짜 파일, 다른 프로그램까지 — 모두 같은 인터페이스로.

### 라인 버퍼링 — `getchar`가 Enter를 기다리는 이유

`getchar()`를 부르면 한 글자가 바로 들어올 것 같지만, 실제로는 사용자가 **Enter를 칠 때까지** 멈춰서 기다린다. 왜일까?

터미널은 사용자가 입력 도중에 **Backspace**로 글자를 지우거나 줄을 수정할 여지를 주려고, 한 줄이 완성될 때까지 입력을 보내지 않고 자기 안에 모아둔다. 이게 **라인 버퍼링(line buffering)** 이다. Enter를 치면 그제야 모은 줄이 `stdin`으로 흘러간다.

그래서 `wd<Enter>`를 치면 `getchar()`는 `'w'`만 받고, **`'d'`와 `'\n'`은 stdin에 그대로 남는다**. 다음 `getchar()`가 그 잔여를 마저 먹는다. 우리는 이걸 **버리는 루프** 한 줄로 처리할 것이다 (코드 보면 나옴).

Stage 3에서는 이 라인 버퍼링을 우회해서 키를 즉시 받는 방법을 다룬다. 오늘은 일단 "Enter를 친다"를 받아들이고 간다.

### ANSI escape — 터미널을 조종하는 작은 명령어

`printf("Hello")`는 글자를 그대로 찍는다. 그런데 `printf("\033[2J")`는 글자 대신 **화면 전체를 지운다**. 어떻게? **터미널이 그렇게 해석하기 때문이다.** 먼저 "터미널"이 정확히 무엇이고 뭘 하는지부터.

#### 잠깐 — "터미널이 글자를 그린다"는 게 정확히 뭔 뜻인가

사실 우리 프로그램은 **글자를 그린 적이 한 번도 없다.** `printf("Hello")`가 하는 일은 stdout 호스에 바이트 5개를 흘려보내는 것 — 그 바이트 자체는 그림이 아니라 그냥 숫자다 (`H = 72, e = 101, l = 108, l = 108, o = 111`).

그 호스 끝이 보통 **터미널 에뮬레이터**(Windows Terminal, iTerm2, gnome-terminal 등)라는 또 다른 프로그램에 연결돼 있다. 터미널이 그 바이트를 받아서:

1. 숫자 72를 폰트에서 찾아 'H' 모양의 픽셀로 그리고
2. 자기 창의 적절한 위치(현재 커서 자리)에 그 픽셀들을 그려 넣고
3. 커서를 한 칸 옆으로 옮겨두기 — 다음 글자가 그 자리에 그려지도록

그래서 우리 눈에 'H'가 보인다. **글자 모양을 아는 것도(폰트), 픽셀로 그리는 것도(렌더링), 화면 어디에 띄울지 정하는 것도(커서) — 다 터미널의 일.**

##### 그럼 터미널이 없으면?

데이터로는 멀쩡히 흐른다. 그림으로는 안 보일 뿐. stdout 호스 끝이 어디 연결되느냐의 문제:
- **터미널에 연결** → 터미널이 그림으로 그려줌 (지금까지의 경우)
- **파일에 연결** (`./snake > out.txt`) → 바이트가 파일에 그대로 쌓임. 화면엔 아무것도 안 떠도 out.txt에 글자 데이터는 정확히 남는다.
- **다른 프로그램에 연결** (`./snake | grep wasd`) → 그 프로그램이 받음.

→ **터미널 = 바이트를 사람 눈에 보이는 글자 그림으로 변환해 주는 별도 프로그램.** OS도 아니고 우리 코드도 아니다.

> 옛날에 "터미널"은 화면 + 키보드만 있는 **물리 장치**였다 (VT100 등). 본체 컴퓨터 한 대에 터미널 여러 대를 케이블로 연결해 함께 썼음. 지금은 그 장치를 소프트웨어로 흉내내는 게 **터미널 에뮬레이터** — 줄여서 "터미널". 이름의 유래.

#### 터미널이 책임지는 일들

위의 '글자 그리기' 외에도 터미널이 맡는 일이 꽤 많다:

1. **글자 그리기** — 받은 바이트를 폰트 글리프로 그려 화면에 픽셀로 표시. 방금 본 그것.
2. **커서 위치 관리** — "다음 글자가 어디에 그려질지". `\033[H` 같은 명령으로 옮길 수 있음.
3. **줄바꿈/스크롤** — 마지막 줄에서 더 출력하면 자동 스크롤. 스크롤백 버퍼도 자기가 관리.
4. **색상·속성** — `\033[31m` 같은 SGR 시퀀스 해석.
5. **입력 라인 버퍼링** — 위에서 다룬 그것. 사용자가 Enter 칠 때까지 키를 모아둠 (canonical mode). raw mode로 바꾸면 즉시 전달 — Stage 3의 `_kbhit/_getch`가 비슷한 효과.
6. **에코(echo)** — 사용자가 누른 키를 자동으로 화면에 표시. 비밀번호 입력 등에선 끔.
7. **시그널 변환** — Ctrl+C → 프로세스에 `SIGINT`, Ctrl+Z → `SIGTSTP`, Ctrl+D → EOF. 터미널이 OS와 협력해서 처리.
8. **창 크기 변화 알림** — 리사이즈 시 프로세스에 `SIGWINCH` 보냄.
9. **대체 스크린 버퍼** — `\033[?1049h`로 별도 화면 모드(vim, less가 쓰는 트릭).
10. **하이퍼링크/이미지** — 최근 터미널(iTerm2, Kitty 등)은 클릭 가능한 링크나 인라인 이미지까지.

#### "상태 기계"의 의미

터미널이 바이트 스트림을 어떻게 처리하는지 분해해 보면:

```
바이트 도착
   ↓
현재 모드는?
   ├─ "글자 모드" (평소)
   │      받은 바이트 → 화면에 글자로 그림
   │      단, \033(ESC)가 오면 → "ESC 모드"로 전환
   │
   ├─ "ESC 모드" (방금 ESC 받음)
   │      [ 가 오면 → "CSI 모드"로 전환
   │
   └─ "CSI 모드" (\033[ 다음)
          숫자/세미콜론 → 파라미터 모음에 쌓고 계속
          알파벳(H, J, m 등) → 명령 실행, 글자 모드로 복귀
```

**현재 상태에 따라 같은 바이트가 다르게 해석되는** 시스템이 상태 기계(state machine). 글자 모드의 `H`는 글자 H, CSI 모드의 `H`는 "커서 홈" 명령. 같은 바이트가 모드 따라 다른 의미를 가진다.

같은 원리가 사방에 있다: 정규표현식 엔진, JSON/HTTP 파서, 게임 캐릭터 상태(걷기/뛰기/공격), TCP 연결 상태(SYN/ESTABLISHED/FIN) — 다 상태 기계의 친척.

#### 비유 한 줄

터미널 ≈ **마크업 렌더러**. 우리 코드가 글자 + 명령 코드(`\033[...`)를 흘려보내면, 터미널은 그걸 해석해서 화면 그림. 브라우저가 HTML을 받아 화면 그리는 것과 같은 추상화.

---

이제 터미널을 조종할 줄 알아야 한다. **이스케이프 시퀀스**(`ESC` 문자 + `[` + 명령)를 만나면 터미널은 그걸 출력 대신 명령으로 해석한다. 그 형식부터 보자.

#### `\033[H` 해부

```
\033  →  ESC 문자 1개 (ASCII 27)
   [  →  CSI(Control Sequence Introducer)의 시작 표시
   H  →  명령 글자 ("커서 홈으로")
```

- **`\033`**: C의 **8진수 표기**(`\033` = 8진수 33 = 십진수 27 = ASCII의 ESC). 대안: `\x1b`(16진수 1B = 27), `\e`(GCC 확장, 비표준이라 이식성↓).
- **`[`**: `ESC` + `[` 조합을 **CSI**라고 부른다. 터미널이 "이제부터 출력이 아니라 명령이야"로 알아듣기 시작.
- **숫자 파라미터**: CSI와 명령 글자 사이에 옴. 여러 개면 `;`로 구분 (예: `\033[3;5H` = 3행 5열로 이동).
- **명령 글자**: 마지막 알파벳 한 글자가 동작을 결정.

일반 형식: `\033[<숫자>;<숫자>;...<명령 글자>`

#### 시퀀스 이어 쓰기

`\033[2J\033[H`는 두 시퀀스를 이어붙인 것 — 화면 지우고 좌상단으로. 파서가 명령 글자(`J`, `H`)에서 시퀀스를 끊어 순서대로 처리하므로 `printf` 한 번에 여러 명령을 묶을 수 있다.

#### 자주 쓰는 명령 (카테고리별)

**커서 이동**
- `\033[H` — 좌상단(1,1)으로 (절대)
- `\033[r;cH` — (r행, c열)로 (절대, 1부터)
- `\033[NA` / `\033[NB` — N줄 **위로** / **아래로** (상대)
- `\033[NC` / `\033[ND` — N칸 **오른쪽** / **왼쪽** (상대)
- `\033[s` / `\033[u` — 현재 위치 저장 / 복원 (일부 터미널만, 이식성↓)

**화면/줄 지우기**
- `\033[2J` — 화면 전체 지움 (스크롤백은 보존)
- `\033[3J` — 화면 + 스크롤백 버퍼까지 지움
- `\033[J` / `\033[1J` — 커서~화면 끝 / 화면 시작~커서
- `\033[2K` — 한 줄 전체 / `\033[K` — 커서~줄 끝

**색상 (SGR — Select Graphic Rendition)**
- `\033[0m` — 모든 속성 리셋 (안전 복귀)
- `\033[31m` ~ `\033[37m` — 글자 색 (빨/초/노/파/마젠타/시안/하양)
- `\033[41m` ~ `\033[47m` — 배경 색
- `\033[1m` — 굵게
- `\033[38;5;Nm` — 256색 글자 (N=0~255)
- `\033[38;2;R;G;Bm` — 트루컬러 글자 (R,G,B = 0~255)

**커서 표시**
- `\033[?25l` / `\033[?25h` — 커서 숨김 / 표시 (게임 그릴 때 깜빡이는 커서 숨기면 깔끔)

**대체 스크린 버퍼** (vim, less, tmux가 쓰는 트릭)
- `\033[?1049h` — 대체 화면 진입 (현재 화면 백업)
- `\033[?1049l` — 원래 화면 복원

이 단계에서는 매 프레임 `\033[2J\033[H`로 **화면을 깨끗이 지우고 처음부터 다시 그리는** 풀-리드로(full-redraw) 패턴을 쓴다. Stage 3에서 더 똑똑한 방법으로 바꾼다.

### 한글이 깨지는 이유 — 문자 인코딩

소스 파일은 보통 UTF-8로 저장된다. 그런데 Windows 콘솔의 기본 출력 코드페이지는 한국 환경에서 CP949(EUC-KR류)다. **같은 바이트가 두 인코딩에서 다른 글자로 해석**되니, 콘솔이 UTF-8 한글을 CP949로 읽으면 깨진 글자가 나온다.

해결: 프로그램 시작 시 `SetConsoleOutputCP(65001)` 한 줄. 65001은 UTF-8의 코드페이지 번호. 이 함수는 Windows API라 `#include <windows.h>`가 필요하다.

#### Windows API와 `windows.h`

OS가 제공하는 함수들의 묶음을 **API**(Application Programming Interface)라고 부른다. 화면에 글자를 띄우고, 파일을 열고, 키 입력을 받고, 콘솔 색을 바꾸는 일은 사용자 프로그램이 직접 못 한다. OS한테 부탁해야 하고, 그 부탁 창구가 API.

Windows는 자기 API를 거대한 C 함수 모음으로 노출한다 — `CreateFile`, `WriteConsole`, `Sleep`, `SetConsoleOutputCP` 등. `windows.h`가 그 선언을 모은 헤더다.

```c
#include <stdio.h>     // 표준 C — 어떤 OS에서도 동작
#include <windows.h>   // Windows API — Windows에서만 동작
```

→ 이 코드는 리눅스/맥에서 컴파일이 안 된다. 그게 **OS 의존성**. `stdio.h`는 어디서나 되지만 `windows.h`는 Windows 전용. Stage 3에서 만나는 `_kbhit`/`_getch`(`conio.h`), `Sleep`(`windows.h`)도 같은 이유로 다른 OS에서 못 쓴다.

#### 코드 페이지란

컴퓨터는 글자를 모른다. **숫자만 안다.** "글자 ↔ 숫자" 매핑 테이블이 필요한데, 이 테이블을 **인코딩** 또는 (Windows 용어로) **코드 페이지(code page)** 라고 부른다.

| 글자 | ASCII | CP949 (한국 Windows) | UTF-8 |
|---|---|---|---|
| `A` | `0x41` | `0x41` | `0x41` |
| `안` | (없음) | `0xBE C8` (2바이트) | `0xEA B0 88` (3바이트) |
| `🐍` | (없음) | (없음) | `0xF0 9F 90 8D` (4바이트) |

같은 "안"이라도 어떤 테이블로 보느냐에 따라 바이트열이 다르다. → **같은 바이트열을 다른 테이블로 해석하면 깨진다.**

Windows 한국어 시스템의 기본 콘솔 코드페이지는 **CP949**. 그런데 요즘 소스 파일은 거의 다 **UTF-8**로 저장된다(VSCode, GitHub, gcc 다 UTF-8 친화). → UTF-8로 저장된 `"이동: wasd"`를 콘솔이 CP949로 해석하면 글자 깨짐.

#### `SetConsoleOutputCP(65001)` 자세히

```c
SetConsoleOutputCP(65001);
// ↑ "이 콘솔의 출력 코드페이지를 65001로 바꿔줘"
//   65001 = Windows가 UTF-8에 부여한 식별자 번호
```

비유: 콘솔이 한글로 적힌 책을 영어 사전으로 읽으려 해서 깨지던 걸, "이 책은 한글이야"라고 알려준 셈.

비슷한 함수가 `SetConsoleCP()` — 이건 **입력** 코드페이지. 우리는 `getchar`(Stage 2) / `_getch`(Stage 3+)로 ASCII 키(w/a/s/d/q)만 받아서 입력 쪽 인코딩은 신경 안 써도 된다.

#### 다른 방법들 (참고)

1. **`chcp 65001`** — 프로그램 실행 *전*에 콘솔(CMD/PowerShell)에서 직접 코드페이지를 바꾼다. 코드에 박을 필요 없지만 매번 해줘야 함.
2. **시스템 설정** — Windows 10/11의 "Beta: Use Unicode UTF-8 for worldwide language support" 옵션. 켜면 시스템 전체 기본이 UTF-8. 다만 일부 옛 프로그램이 깨질 수 있어서 권장 X.
3. **`wprintf` + `wchar_t`** — 와이드 문자(16비트)로 직접 출력. C 표준에 가깝지만 코드가 더 번거롭다.

코드 안에 한 줄 박는 게 가장 휴대성 있는 선택 → 그래서 `SetConsoleOutputCP(65001)`.

## 새 C 문법

### `#include <windows.h>`
- Windows API를 쓰기 위한 헤더. **플랫폼 종속** 코드가 처음 등장.
- `SetConsoleOutputCP`, `Sleep`(Stage 3) 등이 여기서 옴.

### `SetConsoleOutputCP(65001);`
- 콘솔 출력 코드페이지를 UTF-8로 변경. `main` 진입 직후에 호출.

### `while (1) { ... }`
- 조건이 항상 참 → **무한 루프**. 안에서 `break` 또는 `return`으로 빠져나간다.

### `getchar()`
- `stdin`에서 한 글자를 읽는다. 반환 타입은 `char`가 아니라 **`int`**.
  - 왜 `int`? `EOF`(파일 끝, 값은 보통 `-1`)를 "어떤 글자와도 다른 값"으로 표현하려고. `char`로 받으면 EOF와 정상 글자가 충돌할 수 있다.

### `switch (값) { case 'w': ...; break; ... }`
- 값에 따른 분기. 정수 또는 정수로 취급되는 값에만 쓸 수 있다 (`char`도 정수 취급이라 OK).
- **`break`를 빼면 다음 case로 흘러넘친다(fall-through).** 일부러 그렇게 쓰는 경우도 있지만, 처음엔 항상 `break` 붙여두는 게 안전.
- `default:` 절을 넣으면 어느 case에도 안 걸렸을 때 실행.

### `if ( (c = getchar()) != '\n' && c != EOF )`
- C의 관용구. 대입(`=`)이 식 안에서 값을 가진다는 성질을 이용해, "읽으면서 동시에 비교".
- 처음 보면 헷갈리지만 자주 보게 됨. 의미: "한 글자를 읽어 `c`에 넣고, 그게 `\n`도 아니고 `EOF`도 아닌 동안".

### `printf("\033[2J\033[H")`
- 화면 지우기 + 커서 홈. 이스케이프 시퀀스를 그대로 문자열에 넣는다.
- `\033`은 **8진수 표기**의 `033` = 27 = `ESC` 문자.

### 한 문장 if (중괄호 생략)
- `if (player_r < 1) player_r = 1;` — 본문이 한 문장이면 `{}` 생략 가능.
- 두 문장 이상이거나, `else`와 결합되면 `{}` 권장.

## 코드

`01-snake/snake.c`를 다음으로 덮어쓴다.

```c
#include <windows.h>   // (추가) Windows API — SetConsoleOutputCP 사용
#include <stdio.h>     // 표준 입출력 — printf, putchar, getchar

#define ROWS 6
#define COLS 18

int main(void) {
  // (추가) 콘솔 출력 인코딩을 UTF-8로. 한글 깨짐 방지.
  // 코드페이지 65001 = UTF-8. Windows 콘솔은 기본이 CP949(한국)라 안 맞춘다.
  SetConsoleOutputCP(65001);

  printf("Hello, snake!\n");

  // 보드 선언. 내용물은 아래 루프에서 채운다.
  char board[ROWS][COLS];

  // 보드 첫 그림(루프 진입 전 한 번).
  // 이 블록은 곧바로 화면에 찍기까지 함 — Stage 1과 동일한 패턴.
  for (int r = 0; r < ROWS; r++) {
    for (int c = 0; c < COLS; c++) {
      if (r == 0 || r == ROWS - 1 || c == 0 || c == COLS - 1) {
        board[r][c] = '#';
        putchar(board[r][c]);
      } else {
        board[r][c] = '.';
        putchar(board[r][c]);
      }
    }
    putchar('\n');
  }

  // (추가) 플레이어 좌표 — 보드의 정가운데에서 시작.
  // 정수 나눗셈이라 ROWS=6 → 3, COLS=18 → 9.
  int player_r = ROWS / 2;
  int player_c = COLS / 2;

  // (추가) 메인 루프: 키 한 번 받기 → 이동 → 화면 갱신 반복.
  // 종료는 'q' 입력 시 return으로.
  while (1) {
    // ANSI escape: 화면 전체 지우기(\033[2J) + 커서 홈으로(\033[H).
    // 매 프레임 처음부터 다시 그리는 "풀-리드로" 방식.
    printf("\033[2J\033[H");

    // 보드를 다시 채우면서 출력.
    // 매번 board 전체를 새로 칠하는 이유: 플레이어가 지나간 자리를 '.'로 되돌리려고.
    for (int r = 0; r < ROWS; r++) {
      for (int c = 0; c < COLS; c++) {
        if (r == 0 || r == ROWS - 1 || c == 0 || c == COLS - 1) {
          board[r][c] = '#';
        } else {
          board[r][c] = '.';
        }
        // 플레이어 좌표면 'O'로 덮어쓴다.
        // 안쪽 루프에서 검사하는 방식 — 매 칸마다 비교라 살짝 비효율적이지만 코드는 단순.
        board[player_r][player_c] = 'O';
        putchar(board[r][c]);
      }
      putchar('\n');
    }

    printf("이동: wasd, 종료: q\n");

    // 키 한 글자 받기.
    // 주의: getchar는 Enter를 칠 때까지 기다린다 (라인 버퍼링).
    int key = getchar();

    // 남은 글자 + 개행(\n) 버리기.
    // 사용자가 'wd<Enter>'처럼 두 글자를 친 경우 'd'와 '\n'이 stdin에 남는다.
    // 다음 getchar가 그 잔여를 먹지 않도록 미리 비운다.
    // C에서 stdin을 줄 단위로 다룰 때의 관용구.
    int c;
    while ((c = getchar()) != '\n' && c != EOF) {
      // 아무것도 안 함 — 그냥 읽어서 버린다.
    }

    // 키별 분기.
    // 좌표 한 칸 변경. 음수가 될 수 있으니 아래에서 clamp가 필요.
    switch (key) {
      case 'w':                  // 위 = 행이 줄어드는 방향
        player_r--;
        break;
      case 's':                  // 아래
        player_r++;
        break;
      case 'a':                  // 왼쪽 = 열이 줄어드는 방향
        player_c--;
        break;
      case 'd':                  // 오른쪽
        player_c++;
        break;
      case 'q':                  // 종료
        return 0;                // main에서 return → 프로세스 종료
    }

    // 벽 clamp — 항상 switch 뒤에 둔다.
    // 유효 범위: 1 ~ ROWS-2 / 1 ~ COLS-2 (테두리는 # 이므로 그 안쪽).
    // switch에서 좌표를 먼저 바꾼 다음 여기서 가두는 순서가 자연스럽다.
    if (player_r < 1) player_r = 1;
    if (player_r > ROWS - 2) player_r = ROWS - 2;
    if (player_c < 1) player_c = 1;
    if (player_c > COLS - 2) player_c = COLS - 2;
  }

  return 0;
}
```

> **clamp 위치가 왜 switch 뒤인가?** 만약 앞에 두면, 이전 프레임 끝의 좌표를 가두는 셈이라 새 입력으로 빠져나간 좌표는 못 막는다. **"바꾼 다음 가둔다"** 순서가 핵심.

조작:

## 흔히 빠지는 함정

- **한글이 깨진다.** `SetConsoleOutputCP(65001);`을 빠뜨렸거나, `main`의 다른 출력 뒤에 호출했다. 반드시 `main` 시작 직후.
- **`switch`에서 `break`를 빼먹는다.** `'w'`만 눌렀는데 `player_r--`와 `player_r++`가 둘 다 실행되는 등 황당한 동작. `break` 빠뜨림은 워낙 흔해서 컴파일러에 따라 경고를 켜는 옵션(`-Wimplicit-fallthrough`)도 있다.
- **clamp를 switch 앞에 둔다.** 새로 이동한 좌표를 못 막아서 `O`가 벽 밖으로 사라짐.
- **`getchar()`로 글자 하나만 처리하고 끝.** 다음 입력 때 잔여 글자가 한꺼번에 처리되어 `O`가 두 칸 점프하는 듯한 버그. 잔여 비우는 루프가 그래서 있다.
- **`if (key = 'q')`처럼 `=`를 씀.** `==`가 맞음. 컴파일은 통과하므로 가장 짜증나는 종류의 버그.

## 체크 질문

소리 내어 답해보기:

1. 왜 `getchar()`는 Enter를 눌러야 반응할까? **라인 버퍼링**의 의미를 한 문장으로.
2. `stdin`, `stdout`은 파일이 아닌데 왜 파일처럼 다뤄질까? "모든 것은 파일이다" 한 마디로.
3. `switch`에서 `break`를 빼면 어떤 일이 일어나지? 의도적으로 그렇게 쓰는 경우도 있을까?

5. 같은 한글 문자열이 왜 콘솔에선 깨지고 파일에선 멀쩡할까? `SetConsoleOutputCP`가 정확히 무엇을 바꾸지?
6. clamp 코드를 switch **앞**으로 옮기면 무엇이 깨질까?
7. `getchar()`의 반환 타입이 `char`가 아니라 `int`인 이유는?
