# Stage 3 — 게임 루프 (자동 전진)

> 키를 안 누르고 있어도 뱀이 알아서 움직인다. 게임이 게임답게 변하는 순간. 그 한 줄짜리 변화 뒤에 **게임 루프**, **블로킹/논블로킹 I/O**, **차분 렌더링**, **변수 스코프**가 줄줄이 등장한다.

## 오늘의 목표

- 키 입력 없이도 뱀이 매 프레임 한 칸씩 자동 전진한다.
- `wasd`로 **방향만** 바꾼다 (즉시 한 칸이 아니라 "이제부터 이쪽으로").
- `q`를 누르면 즉시 종료 (Enter 없이).
- 매 프레임 전체 보드를 다시 안 그리고 **바뀐 칸만** 갱신한다.

기대 동작:
```
##################
#................#
#......O.........#       ← 가만히 있어도 0.2초마다 O가 오른쪽으로 한 칸씩
#................#
##################
이동: wasd, 종료: q
```

---

## 배경 이야기

### 게임 루프 — 모든 인터랙티브 소프트웨어의 척추

Stage 2까지는 "키 한 번 받으면 한 번 그린다"였다. 사용자가 안 누르면 아무것도 안 일어난다. 게임은 그러면 안 된다. **시간이 흐른다**.

해결책은 너무 단순해서 처음엔 허무하기까지 하다 — **루프를 끊임없이 돌리고, 그 안에서 입력을 체크한다**.

```
while (게임 진행 중) {
  1. 입력 확인     ← 키가 눌렸나? (안 눌렸어도 즉시 진행)
  2. 상태 갱신     ← 뱀 한 칸 전진, clamp 등
  3. 화면 출력     ← 바뀐 부분 그리기
  4. 잠깐 쉬기     ← Sleep(ms) — 프레임 간격
}
```

이게 **게임 루프**다. 그리고 같은 구조가 GUI 앱의 이벤트 루프, 웹 서버의 이벤트 루프, 심지어 OS 스케줄러의 본체에도 들어 있다. 한 번 손으로 짜두면 다른 데서 만나도 알아본다.

### 블로킹 vs 논블로킹 — `getchar`가 안 되는 이유

Stage 2의 `getchar()`를 그대로 루프 안에 두면, 키를 누를 때까지 루프가 **정지**한다. 뱀이 멈춰 있다. `getchar`는 **블로킹(blocking)** 함수다 — 일이 끝날 때까지 스레드를 잡아두는 호출.

게임 루프는 "키가 있으면 처리, 없으면 그냥 넘어가기"가 필요하다. 이걸 **논블로킹(non-blocking)** 이라고 부른다. Windows 콘솔에선 한 쌍의 함수가 그걸 해준다:
- `_kbhit()` — 키가 눌려서 버퍼에 있나? 있으면 1, 없으면 0. **즉시 반환**.
- `_getch()` — 키 한 글자를 즉시 읽음. Enter 안 기다리고, 화면에 에코도 안 함.

이 둘은 `<conio.h>`에 있고, **Windows 전용**이라 함수 이름 앞에 `_` 가 붙어 있다(C 표준이 아니라 처리계 확장이라는 표시).

같은 발상이 사방에 있다:
- 네트워크: `read()` 블로킹 vs `epoll`/`select` 논블로킹.
- JS: 동기 vs `async/await`.
- DB: 동기 쿼리 vs 콜백/Promise.

게임 루프에서 처음 만나는 거지만, 마지막 만남은 아니다.

### 프레임과 FPS — `Sleep`이 곧 속도

`Sleep(ms)`는 OS에 "나 잠깐 재워줘"라고 부탁한다. 이게 한 프레임의 휴식 시간이고, 결국 게임 속도를 결정한다.

```
FPS ≈ 1000 / Sleep_ms
Sleep(200) → 약 5 FPS  ← 우리는 일단 이거. 천천히 보면서 디버깅.
Sleep(16)  → 약 60 FPS ← 부드러운 게임 기본.
```

**왜 그냥 빠르게 루프를 안 돌리고 `Sleep`을 끼우나?** `Sleep` 없이 `while (1) {}`만 두면 CPU 코어 하나를 100% 태운다. **busy-wait**(바쁘게 대기)이라고 한다. 다른 프로그램이 굼떠지고 노트북이 뜨거워진다. `Sleep`은 OS에 CPU를 양보하는 정중한 행동 — 다른 프로세스가 그동안 일할 수 있다. **OS 스케줄링과 협력하는 코드**의 첫 사례.

### 방향 벡터 (dr, dc) — "지금 위치 += 방향"

매 프레임 한 칸 움직이려면 "어느 방향으로?"를 어딘가 저장해 둬야 한다. 두 정수면 충분하다:

```
dr =  0, dc =  1  →  오른쪽으로 한 칸 (행 변화 0, 열 변화 +1)
dr =  0, dc = -1  →  왼쪽
dr = -1, dc =  0  →  위 (행 인덱스가 작아지는 쪽)
dr =  1, dc =  0  →  아래
```

매 프레임 끝에 `player_r += dr; player_c += dc;` 한 줄로 이동. 키 입력은 `dr`, `dc` 값을 **바꿀 뿐** 좌표를 직접 건드리지 않는다. **"무엇을 할지"와 "지금 어디로 가는지"를 분리**하는 작은 리팩터.

이 패턴이 나중에 Stage 4에서 뱀이 길어졌을 때 그대로 확장된다.

### 차분 렌더링 (Dirty Rect) — 바뀐 칸만 다시

Stage 2에선 매 프레임 `\033[2J\033[H`로 화면을 통째로 지우고 전부 다시 찍었다. 작동은 한다. 그런데 화면이 **깜빡거린다**(flicker). 보드 전체를 다시 그리는 IO 비용도 작지 않다.

대안: **바뀐 곳만 갱신**한다.
- 새 위치 `O`를 찍는다.
- 이전 위치를 `.`로 되돌린다.
- 화면을 안 지운다 — 어차피 같은 자리는 같은 글자라 덮어 써도 보기엔 똑같다.

이걸 위해선 **이전 위치를 어딘가 기억**해 둬야 한다. `prev_r`, `prev_c` 변수가 그 역할. "지우려는 것의 좌표를 모르면 못 지운다"는 게 차분 렌더링의 첫째 계명.

화면 지우기 명령도 살짝 바꾼다: `\033[2J\033[H` → `\033[H` (커서 홈만, 화면은 안 지움). 매 프레임 같은 자리에 덮어쓰는 효과.

같은 발상이 ncurses(터미널 UI 라이브러리), 브라우저 렌더링 엔진(어떤 DOM 노드가 dirty인가), 게임 엔진(어떤 객체가 다시 그려져야 하나)에 다 있다.

### 값 복사 vs 별칭 — `int prev_r = player_r;`가 뜻하는 것

`int prev_r = player_r;`라고 쓰면 머릿속에서 "두 변수가 연결됐다"는 인상이 들 수 있다. 아니다. C에서 `=`는 **값을 복사한다**. 이 시점의 `player_r` 값을 읽어 `prev_r`이라는 별개의 변수에 박는다. 이후 `player_r += dr`로 값이 바뀌어도 `prev_r`은 **찍어둔 시점의 값 그대로**.

`board[prev_r][...]`와 `board[player_r][...]`가 "같은 칸"인 것은 **값이 우연히 같을 때뿐**. 한 칸 이동한 다음엔 자연스럽게 서로 다른 칸을 가리킨다.

같은 메모리 칸을 공유하는 "별칭(alias)"이 필요하면 **포인터**가 필요하다 (`int *p = &player_r;`). 그건 Stage 4의 주제. 오늘은 "C의 `=`는 그냥 복사"라는 사실만.

### 변수의 스코프와 수명 — 어디서 선언하느냐가 곧 어디까지 살아 있느냐

`{ ... }` 블록 안에서 선언된 지역변수는 **블록을 벗어나는 순간 사라진다**. 매 루프 반복마다 살아 있어야 하는 값은 **루프 바깥**에서 선언해야 한다.

```c
while (1) {
  int prev_r = player_r;   // ✗ 매 반복마다 새로 생긴다. 직전 프레임 값 소실.
  ...
}
```

vs

```c
int prev_r = player_r;     // ✓ 루프 밖. 한 번만 만들어지고 반복 사이에 살아남는다.
while (1) {
  ...
  prev_r = player_r;       // 단순 값 갱신 (int 안 붙임)
}
```

안쪽에서 `int prev_r`을 다시 선언하면 바깥의 `prev_r`을 가리는 **이름 가림(shadowing)** 까지 일어나 디버깅이 어려워진다. C가 깐깐한 데는 이유가 있다 — 이 규칙으로 스택 메모리를 빡세게 관리하기 때문.

## 새 C 문법

### `#include <conio.h>`
- console I/O 헤더. `_kbhit`, `_getch` 등.
- **Windows MSVC 런타임 전용**. 리눅스/macOS에는 없음. 앞의 `_`는 표준 아님 표시.

### `_kbhit()`
- 키 버퍼에 키가 있는지 검사. 있으면 1(true), 없으면 0. **즉시 반환**, 안 기다림.

### `_getch()`
- Enter 없이 키 한 글자를 즉시 읽음. 화면에 에코 안 됨.
- `_kbhit()`로 키가 있는 걸 확인한 뒤에만 부르는 게 안전.

### `Sleep(ms);`
- 현재 스레드를 ms 밀리초 재움. `<windows.h>`.
- **Windows API**라 함수명이 대문자로 시작 (관습). 리눅스/macOS의 `sleep()`(초 단위 소문자)과는 다른 함수.

### `int dr = 0, dc = 1;`
- 같은 타입 변수 여러 개를 한 줄에 선언 + 초기화. 쉼표로 구분.
- 가독성을 해치진 않으나 변수 수가 늘면 줄 분리 권장.

### `dr = -1, dc = 0;`
- **콤마 연산자**. 여러 식을 한 "문장"으로 나열. `switch`의 case 본문처럼 한 줄에 여러 동작을 넣을 때 보임.
- 단, 일반적으로는 `;`로 분리하는 게 명확. 콤마 연산자는 자제.

### 변수 스코프 규칙
- `{ ... }` 블록이 변수의 생사를 결정. 안에서 선언 = 안에서만 살아 있음.
- 같은 이름을 바깥과 안에 둘 다 선언 가능 — 안쪽 것이 바깥을 **가린다(shadowing)**. 가급적 피한다.

## 코드

`01-snake/snake.c`를 다음으로 덮어쓴다.

```c
#include <windows.h>   // Sleep, SetConsoleOutputCP
#include <stdio.h>     // printf, putchar
#include <conio.h>     // (추가) _kbhit, _getch — Windows 전용 논블로킹 입력

// (수정) 보드 크기. ROWS 6 → 8 — 자동 전진을 관찰하기 좋게.
// #define ROWS 6
#define ROWS 8
#define COLS 18

int main(void) {
  SetConsoleOutputCP(65001);

  printf("Hello, snake!\n");

  char board[ROWS][COLS];

  // (추가) 방향 벡터 — 매 프레임 (player_r, player_c)에 더해진다.
  // 기본 (0, 1) = "오른쪽으로 한 칸씩". 키 입력은 이 두 값만 바꾼다.
  int dr = 0, dc = 1;

  // 보드 초기화는 "루프 진입 전 단 한 번". 차분 렌더링이라 매 프레임 다시 안 칠해도 된다.
  // (삭제) Stage 2에선 안쪽 putchar(board[r][c]) 2줄과 바깥 putchar('\n') 1줄로 동시에 출력했음. 출력은 이제 메인 루프에서.
  for (int r = 0; r < ROWS; r++) {
    for (int c = 0; c < COLS; c++) {
      if (r == 0 || r == ROWS - 1 || c == 0 || c == COLS - 1) {
        board[r][c] = '#';
        // putchar(board[r][c]);
      } else {
        board[r][c] = '.';
        // putchar(board[r][c]);
      }
    }
    // putchar('\n');
  }

  // 플레이어 좌표.
  int player_r = ROWS / 2;
  int player_c = COLS / 2;
  // (추가) 직전 프레임 좌표 — 차분 렌더링에 필요.
  // ★ 반드시 루프 "밖"에서 선언. 안에서 선언하면 매 반복마다 새로 생겨 직전 값을 잃는다.
  int prev_r = player_r;
  int prev_c = player_c;

  while (1) {
    // (수정) 매 프레임 커서만 홈으로. \033[2J(전체 지우기)는 제거 — 깜빡임 + 비효율.
    // printf("\033[2J\033[H");
    printf("\033[H");

    // (추가) === 차분 렌더링: 바뀐 칸만 갱신 ===
    // 직전 머리 자리를 '.'으로 되돌리고, 현재 머리 자리에 'O'를 찍는다.
    // 이 두 줄이 "Dirty Rect" 그 자체.
    board[prev_r][prev_c] = '.';
    board[player_r][player_c] = 'O';

    // (삭제) Stage 2의 매 프레임 보드 재칠하기 이중 for. 차분 렌더링(위 두 줄)이 대체.
    // for (int r = 0; r < ROWS; r++) {
    //   for (int c = 0; c < COLS; c++) {
    //     if (r == 0 || r == ROWS - 1 || c == 0 || c == COLS - 1) {
    //       board[r][c] = '#';
    //     } else {
    //       board[r][c] = '.';
    //     }
    //     board[player_r][player_c] = 'O';
    //     putchar(board[r][c]);
    //   }
    //   putchar('\n');
    // }

    // 보드 전체를 한 번 출력 (커서 홈에서 시작하므로 같은 자리에 덮어쓰는 효과).
    for (int r = 0; r < ROWS; r++) {
      for (int c = 0; c < COLS; c++) {
        putchar(board[r][c]);
      }
      putchar('\n');
    }

    printf("이동: wasd, 종료: q\n");

    // (수정) Stage 2의 블로킹 getchar() + 잔여 비우기 루프 → 논블로킹 _kbhit/_getch.
    // getchar처럼 Enter를 기다리지 않으므로 루프가 멈추지 않는다.
    // int key = getchar();
    // int c;
    // while ((c = getchar()) != '\n' && c != EOF) {
    // }
    if (_kbhit()) {
      int key = _getch();
      // (수정) Stage 2: 좌표 직접 변경(player_r--/++ 등) → 방향 벡터만 변경.
      switch (key) {
        // case 'w': player_r--; break;
        case 'w': dr = -1, dc =  0; break;   // 위
        // case 's': player_r++; break;
        case 's': dr =  1, dc =  0; break;   // 아래
        // case 'a': player_c--; break;
        case 'a': dr =  0, dc = -1; break;   // 왼쪽
        // case 'd': player_c++; break;
        case 'd': dr =  0, dc =  1; break;   // 오른쪽
        case 'q': return 0;                  // 즉시 종료
      }
    }

    // (추가) === 상태 갱신: 이동 직전에 prev 스냅샷, 그 뒤에 이동 ===
    // 순서가 핵심. "현재 위치를 기억" → "한 칸 전진" → 다음 프레임에서 prev로 지움.
    prev_r = player_r;
    prev_c = player_c;
    player_r += dr;
    player_c += dc;

    // 벽 clamp — 이동 후, 다음 프레임 그리기 전에.
    if (player_r < 1)        player_r = 1;
    if (player_r > ROWS - 2) player_r = ROWS - 2;
    if (player_c < 1)        player_c = 1;
    if (player_c > COLS - 2) player_c = COLS - 2;

    // (추가) 디버그용. 좌표 확인용으로 보드 아래에 한 줄 더 찍는다. 익숙해지면 지워도 됨.
    printf("pos: %d, %d\n", player_r, player_c);

    // (추가) 프레임 간격. 200ms = 약 5 FPS. 천천히 보면서 동작 확인용.
    Sleep(200);
  }

  return 0;
}
```

> **prev_r/prev_c 갱신을 switch 안에서 안 하는 이유**: 키를 누른 프레임만 prev가 갱신되고 안 누른 프레임은 갱신이 안 되어 차분 렌더링이 망가진다. **반드시 매 프레임 한 번**, 이동 직전에.

```sh
gcc 01-snake/snake.c -o 01-snake/snake
./01-snake/snake
```

기대 동작:
- 키를 안 눌러도 `O`가 0.2초마다 오른쪽으로 한 칸씩 전진.
- `w/a/s/d`를 누르면 즉시(Enter 없이) 그 방향으로 전환.
- 벽에 닿으면 더 안 나감.
- `q`를 누르면 즉시 종료.

속도가 너무 답답하면 `Sleep(200)`을 `Sleep(100)`이나 `Sleep(50)`으로 바꿔본다.

## 흔히 빠지는 함정

- **`getchar()`를 남겨둠 → 뱀이 멈춤.** 블로킹이라 키 누를 때까지 정지. `_kbhit()` + `_getch()`로 교체.
- **`prev_r`을 while 안에서 `int prev_r = player_r;`로 선언.** 매 반복마다 새 변수가 생겨 직전 값을 잃음. **밖에서 한 번** 선언, 안에서는 `prev_r = player_r;` (int 빼고) 값만 갱신.
- **switch 안에서 `player_r += dr`까지 함.** 그 뒤 루프 끝에서 또 한 번 더해 **한 프레임에 두 칸 이동**. switch는 `dr, dc`만 바꾼다. 좌표 갱신은 루프 끝에서 한 번만.
- **`\033[2J\033[H`를 그대로 둠.** 차분 렌더링인데도 매 프레임 화면을 통째로 지움 → 깜빡임 + 비효율. `\033[H`만.
- **`board[player_r - dr][player_c - dc] = '.';`로 방향 역산.** 방향을 바꾼 프레임이나 clamp가 일어난 프레임에서 엉뚱한 칸을 지움 → O 잔상. `prev_r, prev_c` 스냅샷이 정답.
- **`Sleep`을 너무 줄임.** `Sleep(0)`이나 빼버리면 CPU 100% 점유. 노트북 발열.

## 체크 질문

소리 내어 답해보기:

1. `while(1)`에 `Sleep`이 없으면 CPU는 어떻게 되지? 왜 그런가?
2. `getchar()`와 `_kbhit() + _getch()`의 차이는? "블로킹 / 논블로킹"의 의미를 한 줄로.
3. `Sleep(150)`을 `Sleep(16)`으로 바꾸면 FPS는 어떻게 되지? 공식은?
4. 차분 렌더링(Dirty Rect)의 장단점은? 언제는 풀-리드로가 낫지?
5. `int prev_r`을 while **안**에서 선언하면 왜 이전 위치를 못 기억하나? 변수 스코프와 수명을 직접 설명해보자.
6. `int prev_r = player_r;`를 한 줄 쓰면 두 변수는 같은 메모리를 공유할까? 한 변수가 바뀌면 다른 변수도 바뀔까? "값 복사 vs 별칭"의 의미.
7. `dr, dc` 같은 방향 벡터를 두는 게 왜 좋지? 키 입력에서 좌표를 직접 바꾸는 방식과 비교.
8. `prev_r = player_r;`를 `player_r += dr;` **뒤**에 두면 무엇이 깨질까?
