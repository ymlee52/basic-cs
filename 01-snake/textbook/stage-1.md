# Stage 1 — 정적 보드 그리기

> `#`으로 둘러싸인 사각형 보드를 그린다. 화면엔 단순한 그림이지만, 그 뒤엔 **메모리에 데이터가 어떻게 놓이는가**가 숨어 있다.

## 오늘의 목표

- `#######...`로 테두리를 그리고 안쪽은 `.`으로 채운 사각형 보드를 터미널에 출력한다.
- 2차원 배열이 메모리에 실제로 어떻게 펼쳐지는지 머릿속에 그림으로 그린다.
- **데이터(메모리)와 표현(화면)은 별개**라는 감각을 처음으로 잡는다.

기대 출력:
```
##################
#................#
#................#
#................#
#................#
##################
```

## 배경 이야기 — 2D 배열이 메모리에 놓이는 방식

`char board[6][18]`을 선언하면 머릿속엔 6행 18열짜리 격자가 떠오른다. 그런데 컴퓨터 메모리는 **1차원**이다. 그저 0번지, 1번지, 2번지… 일렬로 늘어선 바이트의 줄.

그럼 격자를 일렬에 어떻게 박아 넣을까? C는 **행 우선(row-major)** 으로 펼쳐 넣는다.

```
메모리 주소:  0   1   2   …   17    18   19   …   35     36 …
실제 칸:    [0][0] [0][1] [0][2] … [0][17] [1][0] [1][1] … [1][17] [2][0] …
            └──────── 0행 (18칸) ────────┘ └──── 1행 ────────┘ └─ 2행 …
```

즉 `board[r][c]`의 주소는 다음과 같이 계산된다:

```
base + (r × COLS) + c
```

#### `base`가 뭐냐

**배열이 메모리에서 시작하는 주소**(base address). 영어 그대로 "기준점".

작은 예시로 `char board[2][3]`(2행 3열, 6칸)이 메모리 어딘가(예시로 `0x1000`이라 치자)에 잡혔다고 하면:

```
주소:       0x1000  0x1001  0x1002  0x1003  0x1004  0x1005
칸:         [0][0]  [0][1]  [0][2]  [1][0]  [1][1]  [1][2]
            └──── 0행 ────┘ └──── 1행 ────┘
```

- **`base` = `0x1000`** — 0행 0열의 주소이자 배열 전체의 시작.
- 검증: `board[1][2]`의 주소 = `0x1000 + (1×3) + 2` = **`0x1005`** ✓
- `r × COLS` = "r행으로 가려면 한 행(COLS칸)씩 r번 건너뛴다". 그 다음 `+ c`로 해당 행 안에서 c열로.

#### `base`를 직접 보고 싶다면

`char board[ROWS][COLS];` 선언 시 컴파일러가 그 함수의 스택 프레임에 영역을 잡고 시작 주소를 `board`라는 이름과 묶는다. 그래서 출력해 볼 수 있음:

```c
printf("%p\n", board);          // base 주소 (예: 0x7ffeefbff610)
printf("%p\n", &board[0][0]);   // 위와 정확히 같음
printf("%p\n", &board[1][0]);   // base + COLS
```

`%p`는 포인터 형식 지정자 — 주소를 16진수로 찍는다. 한 번 실험해 보면 머리에 박힘.

> 매번 실행마다 `base` 값이 달라질 수 있다 (ASLR — 보안용 메모리 주소 무작위화). 절대값보다 **상대 차이**(`&board[1][0] - &board[0][0]` = COLS)가 의미 있음.

#### `sizeof` 곱셈은 어디로 갔나

엄밀한 공식은:
```
주소 = base + ((r × COLS) + c) × sizeof(요소 타입)
```

지금은 `char` 배열이라 `sizeof(char) = 1`이라서 곱셈이 사라져 보일 뿐. 만약 `int board[2][3]`이었다면 `sizeof(int) = 4`라 한 칸씩 4바이트 점프. C는 이걸 **포인터 산술**로 자동 처리 — `board + 1`은 +1바이트가 아니라 **+1행 분량**을 의미한다.

그럼 잠깐 — **`sizeof(char) = 1`은 어디서 온 약속**이고, **`sizeof(int) = 4`는 왜 그런가**? 메모리 기초로 한 발 들어가자.

#### 메모리의 단위 — 바이트와 주소

메모리는 **바이트(byte)** 단위로 잰다. 1바이트 = 8비트 = 0/1이 8개. 한 바이트로 표현 가능한 값은 2⁸ = **256가지**.

메모리는 1차원 바이트 배열이고, 각 칸에 **주소**가 붙어 있다.

```
주소:  0x1000  0x1001  0x1002  0x1003  0x1004  ...
값  : [ 1B   ][ 1B   ][ 1B   ][ 1B   ][ 1B   ] ...
```

CPU는 "주소 X에서 N바이트 읽어와"라는 식으로 동작. 그러니까 **"몇 번지부터"** + **"몇 바이트"** 가 메모리 접근의 기본 두 정보. 앞이 base, 뒤가 sizeof.

#### 자료형 = (크기 + 해석 약속)

C에서 `int x = 42;`라고 쓰면 컴파일러는:
1. 메모리 어딘가에 **4바이트**(보통)를 잡고
2. 그 4바이트에 42의 정수 표현을 박고
3. 그 위치(base 주소)를 `x`라는 이름으로 기억

자료형이 결정하는 것:
- **몇 바이트** 차지하는가 (`sizeof`로 확인 가능)
- 그 바이트들을 **어떻게 해석**할 것인가 (정수? 실수? 문자? 부호 있나?)

같은 4바이트 `0x41 0x42 0x43 0x44`도:
- `int`로 해석 → 1145258561
- `float`로 해석 → 약 12.14
- `char[4]`로 해석 → "ABCD"

→ **메모리에는 타입이 없다.** 코드가 "이 base의 N바이트를 X 타입으로 봐"라고 선언할 뿐.

#### C 자료형 크기 한눈에 (일반적인 64비트 환경)

| 타입 | 크기 (바이트) | 표현 범위 |
|---|---|---|
| `char` | **정확히 1** | -128 ~ 127 (signed) / 0 ~ 255 (unsigned) |
| `short` | 2 | -32,768 ~ 32,767 |
| `int` | 4 | 약 -21억 ~ +21억 |
| `long` | 4 (Windows) / 8 (Linux/Mac) | 환경 의존! |
| `long long` | 8 | 약 ±9×10¹⁸ |
| `float` | 4 | 실수, 7자리 정밀도 |
| `double` | 8 | 실수, 15자리 정밀도 |
| `포인터` (예: `int*`) | 8 (64비트 시스템) | 메모리 주소 한 개 |

> **`char`는 C 표준이 "정확히 1바이트"라고 못박은 유일한 타입.** `sizeof(char) == 1`은 어디서나 보장된다. 그래서 `char board[ROWS][COLS]`는 정확히 `ROWS × COLS` 바이트.  
> 나머지는 표준이 **최소 크기**만 보장하고, 실제 크기는 컴파일러/플랫폼에 따라 다름. 그래서 코드에서 크기가 필요하면 직접 `sizeof`로 묻는다 — Stage 4의 `malloc(sizeof(Cell) * MAX_LEN)`가 이 이유.

##### 왜 크기가 다른가
비트가 많으면 더 많은 값을 표현. 1바이트(256가지) — 글자 한 개 표현하기 좋음. 4바이트(약 43억) — 일반 정수 충분. 8바이트 — 큰 정수, 64비트 메모리 주소. 큰 타입을 항상 쓰면 안전하지만 **메모리 낭비 + 캐시 비효율**.

##### `'#'`이 정확히 1바이트인 이유
`char board[r][c] = '#';`은 1바이트 칸에 `'#'`(ASCII 35) 한 바이트를 박는 것. ASCII는 0~127만 쓰니까 1바이트면 충분. 그래서 보드 한 칸이 1바이트인 게 자연스럽다.

#### base는 사실 모든 변수에 있다

지금까지 "base"를 배열의 시작점으로만 얘기했지만, **모든 변수가 메모리 어딘가의 base를 가진다.** 변수 선언은 "어떤 타입 크기만큼 메모리를 잡고 그 시작 주소를 이름과 묶기"의 다른 말.

```c
char  ch  = 'A';   // base_ch  부터 1바이트
int   n   = 42;    // base_n   부터 4바이트
char  arr[10];     // base_arr 부터 10바이트 (= 10 × sizeof(char))
int   nums[10];    // base_nums부터 40바이트 (= 10 × sizeof(int))
```

이걸 `&ch`, `&n`, `arr`, `nums`처럼 **주소 가져오기 연산자 `&`** (또는 배열 이름 자체)로 직접 볼 수 있다. `%p`로 찍으면 그 base가 보인다.

##### 그래서 2D 배열 공식이 자연스러워진다

```
&board[r][c] = base + ((r × COLS) + c) × sizeof(요소 타입)
```

세 조각이 각자 의미를 가짐:
- **base** — 배열이 시작하는 곳 (변수 선언이 결정)
- **(r × COLS) + c** — 1차원으로 펼친 인덱스 (행 우선 레이아웃이 결정)
- **× sizeof(요소 타입)** — 한 칸이 몇 바이트인지 (자료형이 결정)

세 가지 정보 — **어디서, 얼마나 떨어진, 어떤 단위로** — 가 합쳐져야 비로소 "메모리 어디"가 나온다. 이 공식이 다음 단계에 등장하는 모든 포인터 연산의 뼈대.

#### 왜 알아둘 가치가 있나

- **버퍼 오버플로의 원리**: `board[6][0]`을 접근하면 공식대로 `base + 6×COLS + 0`을 건드림 → 보드 다음에 잡힌 다른 변수의 메모리를 침범. C는 막아주지 않음. 런타임에 어디로 튈지 모름.
- **캐시 친화성**: 큰 행렬을 **행 방향**으로 도는 게 열 방향보다 빠르다. 같은 행은 메모리에서 붙어 있어 CPU 캐시에 한 번에 올라옴. 열 방향은 매번 `COLS`칸씩 점프해 캐시 미스가 자주.
- **포인터로 가는 다리**: 배열 인덱싱이 결국 "주소 계산 + 역참조"라는 사실이 Stage 4의 `body[i] ≡ *(body + i)`로 이어진다.

지금은 사각형 한 번 그리는 것뿐이지만, 이 한 줄(`board[r][c] = '#'`)에서 일어나는 일은 이만큼 깊다.

## 데이터 ≠ 표현 — 게임 코드의 큰 패턴 첫 등장

`board[r][c] = '#';`는 **메모리에 글자 한 개를 박는** 명령이다. 화면엔 아무것도 안 뜬다. 화면에 나타나려면 따로 `putchar(board[r][c])`로 **stdout으로 보내야** 비로소 보인다.

```
상태(state)               렌더(render)
board[r][c] = '#'    →    putchar(board[r][c])
   ↑                          ↑
RAM에 박힘 (안 보임)     화면에 뜸 (보임)
```

지금은 두 줄이 바로 옆에 붙어 있어서 차이가 안 느껴진다. 하지만 Stage 3에서 게임 루프가 등장하면 **상태 업데이트와 렌더링이 다른 시점에 일어나는 패턴**이 명확해진다. 모든 인터랙티브 소프트웨어의 척추가 이 분리다 — 게임, GUI, 웹 모두.

## 새 C 문법

### `#define ROWS 6`
- **매크로 상수**. 변수가 아니다. 전처리기가 컴파일 전에 소스의 `ROWS`라는 글자를 모두 `6`으로 **단순 치환**한다.
- 엑셀의 Ctrl + H (찾기 및 바꾸기) 생각하면 됨. 컴파일 전에 소스 파일(.c)을 **전처리, Preprocessing**하는 과정을 한 번 거치는데, 이때 그냥 ROWS라는 글자를 보면 다 6으로 무식하게 바꿈
- 그래서 메모리도 안 차지하고, 실행 중에 바꿀 수도 없다.
- 관습적으로 **대문자**로 쓴다 (일반 변수와 구분).
- 배열 크기처럼 "컴파일 시점에 알아야 하는 값"에 자주 쓴다.

### `char board[ROWS][COLS];`
- 2차원 배열 선언. `[행][열]` 순서.
- `char`이므로 한 칸 = 1바이트. 전체 크기 = `ROWS × COLS × 1` = 108바이트.
- 선언만 하면 메모리는 **쓰레기 값**으로 시작한다. 직접 채우기 전엔 그 안에 뭐가 있을지 모름.

### `for (int r = 0; r < ROWS; r++) { ... }`
- C의 가장 일반적인 반복문.
- 세 부분: **초기화**(`int r = 0`), **조건**(`r < ROWS`), **증감**(`r++`).
- `int r`을 for 안에서 선언하면 그 변수는 **이 for 안에서만 살아 있다**. (변수 스코프 — Stage 3에서 다시 만남.)
- `r++`는 `r = r + 1`의 축약. `r--`는 `r = r - 1`.

### `if (조건) { ... } else { ... }`
- 조건은 `||`(OR), `&&`(AND), `!`(NOT), `==`(같다), `!=`(다르다) 등으로 조합.
- 본문이 한 문장이면 `{}` 생략 가능하지만, 처음엔 항상 붙여두는 게 안전. `if`/`else` 짝이 헷갈리는 버그 방지.

### `'#'` vs `"#"`
- `'#'` (작은따옴표) — **문자 한 개**. C에서 타입은 `char`(또는 `int`). 1바이트.
- `"#"` (큰따옴표) — **문자열**. `#` 다음에 보이지 않는 종결자 `\0`이 붙은 2바이트짜리.
- 둘은 **다른 타입**이라 서로 바꿔 쓸 수 없다. `board[r][c] = "#";`은 컴파일 에러.

### `putchar(c)`
- 문자 한 개를 표준 출력으로 보낸다. `printf("%c", c)`와 같다.
- `printf`보다 가볍다. 한 글자 단위로 찍을 땐 이게 더 자연스러움.

## 코드

`01-snake/snake.c`를 다음으로 채운다. (Stage 0의 5줄짜리는 통째로 덮어쓴다.)

```c
#include <stdio.h>

// (추가) 매크로 상수: 전처리기가 컴파일 전에 "글자 치환"으로 박아넣음.
// 변수가 아니라 그냥 이름표라, 메모리도 안 먹고 런타임에 바꿀 수도 없다.
// 보드 크기처럼 "컴파일 시점에 알아야 하는 값"에 적합.
#define ROWS 6
#define COLS 18

int main(void) {
  // (수정) 끝에 \n 붙임 — 다음 줄부터 보드가 깔끔하게 시작되도록.
  // printf("Hello, snake!");
  printf("Hello, snake!\n");

  // (추가) 2차원 char 배열. 메모리에는 ROWS*COLS 바이트가 일렬로 잡힌다.
  // 선언만으로는 내용물은 "쓰레기 값" — 뒤에서 직접 채워야 의미가 생긴다.
  char board[ROWS][COLS];

  // (추가) 이중 for로 테두리는 '#', 안쪽은 '.'을 board에 박으면서 동시에 putchar로 출력.
  // 바깥 루프: 행(r) 0 ~ ROWS-1
  for (int r = 0; r < ROWS; r++) {
    // 안쪽 루프: 열(c) 0 ~ COLS-1
    for (int c = 0; c < COLS; c++) {
      // 테두리 판정: 첫 행, 마지막 행, 첫 열, 마지막 열 중 하나면 '#'.
      // 부등식이 아닌 등호로 비교하는 게 핵심 — "딱 가장자리만".
      if (r == 0 || r == ROWS - 1 || c == 0 || c == COLS - 1) {
        board[r][c] = '#';      // 1) 데이터에 저장 (메모리)
        putchar(board[r][c]);   // 2) 같은 글자를 표준 출력으로 (화면)
      } else {
        board[r][c] = '.';
        putchar(board[r][c]);
      }
    }
    // 한 행이 끝났으니 줄바꿈. 이게 없으면 모든 칸이 한 줄에 이어붙음.
    // 위치 주의: '바깥 for 안 / 안쪽 for 밖' — 행마다 한 번만 호출되어야 함.
    putchar('\n');
  }

  return 0;
}
```

> **왜 `board[r][c] = ...`와 `putchar(...)`를 따로 두 줄로 쓰는가?**
> 한 줄로 `putchar('#')`만 해도 화면엔 똑같이 나온다. 하지만 그러면 보드가 **메모리에 안 남는다**. Stage 2부터는 보드를 기억하고 있다가 다시 그려야 하므로, 지금부터 "데이터에 저장 → 그 데이터를 보여줌" 패턴을 들여두는 것.

## 흔히 빠지는 함정

- **`putchar('\n')` 위치를 안쪽 for에 넣는다.** → 칸마다 줄바꿈이 되어 글자가 한 줄에 한 개씩 세로로 나옴. **바깥 for 안, 안쪽 for 밖**에 둬야 행마다 한 번.
- **`board[r][c] = "#"`로 큰따옴표 사용.** → 컴파일 에러. 작은따옴표 `'#'` 사용.
- **`r <= ROWS`로 조건을 잘못 씀.** → `board[6][...]`을 건드림(배열은 `[0]~[5]`까지만 유효). 운 좋으면 그냥 잘못된 글자가 찍히고, 운 나쁘면 세그폴트. **`<` 와 `<=` 의 차이는 항상 신중히.**
- **`if (r = 0)`**처럼 `=`와 `==`를 혼동. → 그 줄은 "0을 대입하면서 결과 0(거짓)으로 평가"되어 조용히 다른 동작. 컴파일은 통과하므로 가장 못된 버그.

## 체크 질문

소리 내어 답해보기:

1. `board[3][5]`의 메모리 주소는 어떻게 계산되지? (base 주소를 0이라 가정, `COLS = 18`)
2. `board[6][0]`에 접근하면 무슨 일이 일어날까? (배열은 `[6][18]` 크기)
3. `#define ROWS 6`은 일반 변수 `int ROWS = 6;`과 어떻게 다른가? 메모리에 ROWS라는 라벨이 남나?
4. `'#'`과 `"#"`은 메모리에서 각각 몇 바이트지? 왜 다르지?
5. `board[r][c] = '#';`만 쓰고 `putchar`를 빼면 화면엔 뭐가 보일까? 데이터는 어떻게 되지?
6. `putchar('\n')`을 안쪽 for 안에 넣으면 출력은 어떻게 망가질까?
