# 01-snake — 학습 노트

터미널에서 화살표키로 조작하는 스네이크 게임을 만들면서 C 문법과 CS 개념을 익힌다.

## 진행 현황

| Stage | 내용 | CS 개념 | 상태 |
|---|---|---|---|
| 0 | Hello World 컴파일 | 컴파일 → 링크 → 실행 흐름 | ✅ |
| 1 | 정적 보드 그리기 | 2D 배열, 메모리 레이아웃 | ✅ |
| 2 | 키 한 번 받아 이동 | 표준 입출력, 버퍼링, ANSI escape | ✅ |
| 3 | 게임 루프 (자동 전진) | 무한 루프, 틱(tick), Sleep vs busy-wait | ✅ |
| 4 | 사과 먹고 길어지기 | 구조체, malloc/free, 큐 | ⬜ |
| 5 | 충돌·게임오버·점수 | 상태 머신, 난수 | ⬜ |
| 6 (선택) | 배열 → 연결 리스트 리팩터 | 포인터, 연결 리스트 | ⬜ |

## 단계별 체크 질문

- **Stage 0**
  - `.c` → `.exe` 사이에 어떤 단계를 거치지?
  - 프로그램이 끝나면 왜 콘솔 창이 닫히지? 터미널에서 실행하면 왜 안 닫힐까?
- **Stage 1**
  - `board[3][5]`의 메모리 주소는 어떻게 계산되지? (base = 0, COLS = 18)
  - `board[6][0]`을 접근하면 무슨 일이? (배열 크기는 `[6][18]`)
  - `#define ROWS 6`은 일반 변수와 어떻게 다른가?
- **Stage 2**
  - 왜 `getchar()`는 Enter를 눌러야 반응할까? (라인 버퍼링)
  - `stdin`, `stdout`은 파일도 아닌데 왜 파일처럼 다뤄질까? (유닉스 철학)
  - `switch`에서 `break`를 빼면 무슨 일이? (fall-through)
  - 같은 "이동" 문자열이 왜 파일과 콘솔에서 다르게 보일까? (문자 인코딩)
- **Stage 3**
  - `while(1)`에 `Sleep`이 없으면 CPU는 어떻게 되지?
  - `getchar()`와 `_kbhit() + _getch()`의 차이는? (블로킹 vs 논블로킹)
  - `Sleep(150)`을 `Sleep(16)`으로 바꾸면 FPS는 어떻게 되지?
  - 차분 렌더링(Dirty Rect)의 장단점은? 언제 쓰면 안 되지?
  - `int prev_r`을 while 안에서 선언하면 왜 이전 위치를 못 기억하나? (변수 스코프와 수명)
- **Stage 4**: `malloc`한 메모리를 `free` 안 하면 무슨 일이?
- **Stage 5**: `srand`를 안 부르면 `rand()`가 매번 같은 이유는?
- **Stage 6**: 뱀 몸통은 배열 vs 연결 리스트 중 어느 쪽이 자연스럽지? 왜?

## 처음 본 C 문법

> 단계마다 새로 만난 문법을 한 줄씩.

- **Stage 0**
  - `#include <stdio.h>` — 전처리기 지시문. 표준 입출력 함수 쓰려면 필요.
  - `int main(void) { ... }` — 프로그램의 진입점. 정수를 반환, 인자 없음.
  - `printf("...\n")` — 화면 출력. `\n`은 줄바꿈.
  - `return 0;` — 정상 종료. 0이 아니면 에러 코드. 모든 문장은 `;`로 끝.

- **Stage 1**
  - `#define ROWS 6` — 매크로 상수. 전처리 단계에서 단순 치환되며, 관습적으로 대문자.
  - `char board[ROWS][COLS];` — 2D 배열 선언. 자료형 + 이름 + 차원 + `;`.
  - `for (int r = 0; r < ROWS; r++) { ... }` — for 루프 (초기화 / 조건 / 증감). 안에서 선언한 `int r`은 그 루프에서만 살아있음.
  - `if (조건) { ... } else { ... }` — 조건 분기. 조건은 `||`(OR), `&&`(AND)로 묶을 수 있음.
  - `'#'` (작은따옴표) — 문자 1개. `"#"` (큰따옴표) — 문자열. C에서 둘은 다른 타입.
  - `putchar(c)` — 문자 1개를 화면에 출력. `printf("%c", c)`와 동일.

- **Stage 2**
  - `int x = 값;` — 변수 선언과 초기화. `x = x + 1`을 `x++`로 축약 가능.
  - `while (조건) { ... }` — 조건이 참인 동안 반복. `while (1)`은 무한 루프, `break`로 탈출.
  - `switch (값) { case 'w': ...; break; ... }` — 값에 따른 분기. `break` 빼면 다음 case로 흘러넘침(fall-through).
  - `getchar()` — 키 한 개 읽음. 반환 타입은 `char`가 아니라 **`int`** (EOF 표현 위해).
  - `printf("\033[2J\033[H")` — ANSI escape 시퀀스. 화면 지우기 + 커서 홈.
  - `%d` — `printf`의 정수 형식 지정자. `%c`(문자), `%s`(문자열)와 비교.
  - `if (조건) 한문장;` — 중괄호 생략 가능 (한 문장일 때만). 여러 줄이면 `{}` 필수.
  - `#include <windows.h>` — Windows API 호출용 헤더. 플랫폼 종속 코드 진입.
  - `SetConsoleOutputCP(65001);` — 콘솔 출력 코드페이지를 UTF-8로 바꿈. `main` 맨 처음에.

- **Stage 3**
  - `#include <conio.h>` — console I/O. `_kbhit`, `_getch` 등 제공. Windows MSVC 런타임 전용. 앞의 `_`는 "C 표준 아님, 처리계 확장" 표시.
  - `_kbhit()` — 키가 눌려 있으면 1, 아니면 0 **즉시** 반환. 대기 X. 논블로킹 판별.
  - `_getch()` — Enter 없이 키 한 개를 즉시 읽음. 입력한 문자는 화면에 에코되지 않음.
  - `Sleep(ms);` — 현재 스레드를 밀리초 단위로 재움. `<windows.h>`의 Windows API (앞 글자 대문자 관습).
  - `int dr = 0, dc = 1;` — 한 줄에 같은 타입 변수 여러 개 선언/초기화.
  - `dr = -1, dc = 0;` — 콤마 연산자. 여러 식을 한 "문장"에 나열. 가독성 떨어지므로 `;` 분리 권장.
  - 변수 스코프(scope): `{ }` 블록 안에 선언된 지역변수는 블록이 끝나면 **소멸**. 루프 안에서 매번 새로 선언하면 직전 반복 값은 사라짐. 지속돼야 하는 상태는 루프 **밖**에 선언해야 함.

## 깨달은 CS 개념

> 단계마다 "아 이거구나" 한 것을 짧게.

- **Stage 0**
  - `.c` → `.exe`는 한 번에 되는 게 아니라 **전처리 → 컴파일 → 어셈블 → 링크** 단계를 거친다. `gcc`가 이걸 한 줄로 묶어줄 뿐.
  - **프로세스의 수명 = 콘솔 창의 수명.** `main`이 `return`하면 프로세스가 끝나고, Windows가 자동으로 콘솔 창을 닫는다. 그래서 exe를 더블클릭하면 창이 깜빡이고 사라진다.
  - 터미널에서 `./파일명`으로 실행하면 **이미 열려있는 터미널 프로세스**가 출력을 받아주기 때문에 창이 안 닫힌다.

- **Stage 1**
  - **2D 배열도 메모리에선 1차원이다.** `char board[6][18]`은 `[0행 0열][0행 1열]...[0행 17열][1행 0열]...` 순으로 한 줄로 펼쳐 저장됨 (행 우선, row-major). `board[r][c]`의 주소 = `base + r * COLS + c`.
  - **데이터(메모리)와 표현(화면)은 분리되어 있다.** `board[r][c] = '#'`은 RAM에 글자를 넣을 뿐, 화면엔 아무것도 안 뜸. `putchar`/`printf`로 명시적으로 출력해야 비로소 stdout으로 보내짐. → 게임 루프에서 "상태 갱신"과 "렌더링"이 분리되는 패턴의 시작.
  - **매크로(`#define`)는 변수가 아니다.** 컴파일 시작 전에 전처리기가 단순 치환만 함. 그래서 메모리도 안 차지하고, 런타임에 바꿀 수도 없음.

- **Stage 2**
  - **표준 입출력 스트림은 파일처럼 다뤄진다.** 모든 프로세스는 태어날 때 `stdin`(0), `stdout`(1), `stderr`(2) 세 개를 받음. 키보드/화면/파일/파이프가 같은 인터페이스로 취급되는 게 유닉스 철학. 그래서 `./snake > out.txt`로 출력을 파일에 돌려도 코드는 안 바뀜.
  - **라인 버퍼링.** 터미널은 사용자가 Enter를 치기 전까지 키를 안 넘겨줌 (수정할 여지 주려고). 그래서 `getchar()`가 Enter를 기다리는 것처럼 보이는 것. Stage 3에서 이 버퍼링을 우회함.
  - **ANSI escape 코드.** `\033[...` 같은 제어 시퀀스를 터미널이 "출력 대신 동작"으로 해석. 커서 이동, 화면 지우기, 색상 등. 터미널이란 게 단순히 글자 찍는 창이 아니라 **작은 상태 기계**라는 증거.
  - **문자 인코딩은 "글자 = 바이트"가 아니다.** 파일이 UTF-8로 저장돼도 콘솔이 CP949로 해석하면 같은 바이트가 다른 글자가 됨. `SetConsoleOutputCP(65001)`로 콘솔 쪽 인코딩을 UTF-8로 맞춰주면 해결. → 플랫폼 종속 API를 처음 호출해봄.

- **Stage 3**
  - **게임 루프는 모든 인터랙티브 소프트웨어의 뼈대.** `Input → Update → Render → Sleep`의 순환. GUI 앱, 웹 서버(event loop), OS 스케줄러까지 이 구조의 변형. 사용자 입력을 기다리며 "멈춰 있는" 게 아니라 **쉴 새 없이 돌면서** 입력을 주기적으로 체크한다.
  - **블로킹 vs 논블로킹.** `getchar()`는 입력이 올 때까지 스레드를 멈춤(블로킹). 게임 루프에 이걸 넣으면 "키 안 누르면 정지"가 됨. `_kbhit() + _getch()` 조합은 "지금 당장 있는가"만 확인하고 바로 넘어감(논블로킹) → 루프가 스스로 굴러감. 서버의 `read()` vs `epoll`, JS의 sync vs async도 같은 개념.
  - **프레임과 FPS.** `Sleep(ms)`의 값이 곧 프레임 시간. `FPS ≈ 1000 / ms`. 속도를 코드가 아니라 **데이터**(숫자 하나)로 분리해두면 튜닝이 쉬움 — 게임에서 "난이도" = 이 숫자를 바꾸는 것.
  - **Sleep vs busy-wait.** `Sleep`은 OS에게 "나 재워줘" 라고 양보 → 다른 프로세스에 CPU 넘김(착한 시민). `while (시간 안 됨) {}` 같은 busy loop는 CPU 코어 하나를 100% 태움(나쁜 시민). OS 스케줄링과 협력하는 코드를 짜야 한다.
  - **상태와 렌더는 별개 단계.** `player_r` 값을 바꾸는 것(상태)과 그걸 화면에 반영하는 것(렌더)은 분리되어 있다. 상태만 바뀌고 board에 안 찍히면 화면에 안 보임. Stage 1에서 배운 "데이터 ≠ 표현"의 확장.
  - **차분 렌더링(Dirty Rect).** 매 프레임 전체를 다시 그리는(Full-redraw) 대신, "바뀐 자리만" 건드리는 최적화. 빠르지만 **이전 상태를 명시적으로 기억**해야 함(`prev_r`, `prev_c`). 없애고 싶은 것의 위치를 모르면 못 지우니까. 터미널 라이브러리(ncurses)와 브라우저 렌더러의 공통 개념.
  - **변수의 스코프와 수명.** C는 `{ }` 블록을 벗어나면 지역변수를 파괴한다. 매 루프 반복마다 살아남아야 하는 값은 **루프 밖**에 선언해야 함. 안쪽에 다시 `int x`를 선언하면 **이름 가림(shadowing)** 이 일어나 바깥 값과 단절됨. JS/Python은 느슨하지만 C는 이 규칙이 엄격하다 — 스택 메모리를 빡세게 관리하기 때문.
  - **ANSI escape 확장: `\033[H` vs `\033[2J\033[H`.** 전자는 커서만 맨 위로, 후자는 화면을 "지우고" 맨 위로. 차분 렌더링에서는 화면을 지워버리면 안 되므로 `\033[H`만 쓰는 게 정석. 덤으로 깜빡임(flicker)도 줄어든다.
  - **값 복사 vs 별칭(alias).** C의 `int prev_r = player_r;`는 **현재 값을 복사**할 뿐, 두 변수를 "연결"하지 않는다. 이후 `player_r`이 바뀌어도 `prev_r`은 찍어둔 시점의 값 그대로. 같은 메모리를 공유하려면 **포인터**(`int *p = &player_r;`)가 필요하다. 그래서 `board[prev_r][...]`와 `board[player_r][...]`가 "같은 칸"을 가리키는 건 **값이 같을 때뿐**이고, 이동(`player_r += dr`) 뒤엔 자연스럽게 서로 다른 칸이 된다. → Stage 4의 포인터/malloc으로 이어짐.

## 삽질 기록

> 에러 메시지 / 오타 / 세그폴트 → 어떻게 해결했나.

- **Stage 0**: exe 파일을 더블클릭했더니 창이 떴다 사라짐. → 원인: 프로세스가 끝나면 창도 닫힘. → 해결: 터미널을 먼저 열고 거기서 `./01-snake/snake` 실행.
- **Stage 1**: 이중 반복문 내 putchar('\n'); 위치 헷갈림. → 어디에서 줄바꿈해줘야 할까? → 행 반복문 안, 열 반복문 밑에 배치.
- **Stage 2**: 안내문 한글이 깨져서 나옴. → 원인: 소스 파일은 UTF-8인데 Windows 콘솔은 기본 CP949로 해석. → 해결: `#include <windows.h>` 추가 후 `main` 시작에서 `SetConsoleOutputCP(65001);`로 콘솔을 UTF-8로 전환.
- **Stage 3 (1)**: `getchar()`를 루프에 남겨두니 뱀이 정지, 주석처리하면 자동 전진. → 원인: `getchar()`는 **블로킹** 함수라 입력이 올 때까지 루프가 멈춤. → 해결: `_kbhit()`로 키 여부만 확인하고, 눌렸을 때만 `_getch()`로 읽는 논블로킹 패턴으로 교체.
- **Stage 3 (2)**: 차분 렌더링 시도. 지우기 위치를 `board[player_r - dr][player_c - dc] = '.'`로 "방향 역산"해서 계산 → 방향 전환/clamp 발생 시 엉뚱한 칸을 지워 O 잔상이 남음. → 원인: 이전 위치를 방향에서 추정하려 하면 방향이 바뀐 프레임에서 틀어짐. → 해결: `prev_r, prev_c` 변수를 **이동 직전에 스냅샷**해서 명시적으로 기억.
- **Stage 3 (3)**: `int prev_r = player_r;`를 while 루프 **안**에 선언해서 매 프레임 `prev == player`가 됨 → 지난 프레임 위치 소실, O 잔상 계속 남음. → 원인: `{ }` 블록 끝에서 지역변수 소멸, 다음 반복에 `int` 재선언으로 초기화됨(스코프 + 이름 가림). → 해결: 루프 **밖**에서 `int prev_r = player_r;` 한 번 선언, 루프 안에서는 `prev_r = player_r;` (int 없이) 값만 갱신.
- **Stage 3 (4)**: switch 안에서 `player_r += dr` 해두고 루프 끝에도 `player_r += dr` 남겨둠 → 키 누른 프레임만 2칸씩 이동. → 해결: 이동은 프레임 끝에서 **한 번만**. switch는 `dr, dc` 값만 바꾼다.
- **Stage 3 (5)**: `\033[2J\033[H`를 그대로 쓰니 차분 렌더링인데도 화면 전체가 지워짐(비효율 + 깜빡임). → 해결: 차분 모드에서는 `\033[H`(커서 홈)만 쓰고 `2J`(화면 지우기)는 제거.

### 🐛 `free(body)` 도달 불가 이슈
- 현재 `while(1)` → `case 'q': return 0;` 로만 종료 → `while` **밖**의 `free(body);` 가 실행되지 않음.
- 고치는 법 (택일):
  - `case 'q'` 안에서 `free(body); return 0;` 하기
  - `int running = 1;` 플래그 써서 `while (running)` 로 바꾸고 `case 'q'`에서 `running = 0;`
- 실무 팁: OS가 프로세스 종료 시 힙을 회수하므로 이 프로그램은 실제로는 문제 없지만, **`malloc` 짝에는 `free`** 원칙은 지키는 게 습관.

### 🔤 Stage 4에서 새로 본 C 문법

- `#include <stdlib.h>` — `malloc`, `free`, `rand`, `srand`.
- `#include <time.h>` — `time()`.
- `typedef struct { int r; int c; } Cell;` — 여러 필드를 한 타입으로 묶기 + 별명(typedef). 이후 `struct` 키워드 없이 `Cell x;` 로 사용.
- `.r`, `.c` — 구조체 멤버 접근 (점 연산자).
- `Cell a = b;` — 같은 타입 구조체끼리 통째 복사(`=` 가 모든 필드 한 번에 복사).
- `Cell *body;` — **포인터** 선언. "Cell을 가리키는 주소" 변수. `Cell body;`(본체)와 다름.
- `malloc(sizeof(Cell) * N)` — 힙에서 N개 분량 바이트 빌려오기. 반환값은 주소(포인터).
- `sizeof(타입)` — 타입의 바이트 수를 컴파일러가 계산해줌. 플랫폼 독립적으로 쓰려면 직접 숫자 적지 말고 `sizeof` 사용.
- `body[i]` ≡ `*(body + i)` — 포인터에 `[]` 접근하면 배열처럼 동작 (C의 핵심 규칙).
- `body[i].r` — 포인터 인덱싱 → 구조체 → 멤버 접근까지 한 줄에.
- `free(ptr);` — `malloc`한 메모리를 OS에 반환. 빼먹으면 누수.
- `#define MAX_LEN (ROWS * COLS)` — 매크로 안에서 식 사용 가능. 괄호 쳐두는 게 안전.

### 🧠 Stage 4에서 다룰 CS 개념 요약 (다음에 깊이 설명)

- **구조체(struct)**: 관련 필드를 하나의 타입으로 묶음. `int r[], int c[]` 두 배열 대신 `Cell body[]` 하나.
- **포인터 vs 값**: `Cell body;`(본체, 스택에 한 개) vs `Cell *body;`(주소, 본체는 힙 어딘가에). C의 "값 vs 참조" 구분의 시작.
- **동적 메모리(malloc/free)**: 컴파일 타임에 크기 모를 때 힙에서 빌려오는 방식. 자동 회수 안 되므로 수동 `free` 필요.
- **스택 vs 힙**:
  - 스택 = 작음, 자동 관리, 빠름. 지역변수·고정 크기 배열.
  - 힙 = 큼, 수동 관리, 느림. `malloc`한 메모리.
- **배열-포인터 동치성**: `body[i]` == `*(body + i)`. 포인터 산술은 타입 크기만큼 증가(`+i` 는 `i * sizeof(Cell)` 바이트).
- **큐(Queue, FIFO)**: 뱀 몸통의 자연스러운 모델 — 머리 enqueue, 꼬리 dequeue. Stage 3의 `prev_r/c`는 길이 1 큐였음.
- **shift 루프 방향**: 뒤에서 앞으로 돌지 않으면 덮어쓰기로 데이터 손실. 큐 구현의 고전 함정.
- **메모리 누수(memory leak)**: `free` 누락 → 힙에 짐이 쌓임. 짧은 프로그램은 OS가 봐주지만 장기 실행 서버에서 치명적.
