# C로 배우는 컴퓨터공학 — 학습 로드맵

> 목표: C로 작은 프로젝트를 만들면서 **컴퓨터공학 핵심 개념(메모리·OS·자료구조·네트워크)을 손으로 만져본다.**
>
> 방식: **재미있는 결과물**(게임·도구·에뮬레이터)을 만들되, 그 안에 끼어드는 CS 개념 하나하나를 체화. C 문법은 부수적.

---

## 왜 이 방식인가

- **CS 개념은 글로 읽으면 안 남는다.** "스택은 LIFO다"를 외워봐야 직접 스택을 손으로 짜본 사람과는 차이가 큼.
- **C는 컴퓨터의 속살을 가장 적게 가린다.** 메모리 주소·바이트·포인터가 코드에 그대로 보임. 다른 언어로는 못 보는 것들.
- **프로젝트는 "결과물이 보이는 것"으로.** 게임 / 도구 / 에뮬레이터처럼 동작이 눈에 보여야 동기부여 유지.
- **진도보다 깊이.** 한 프로젝트 끝낼 때마다 "이 안에서 어떤 CS 개념을 만났지?"를 정리하는 게 핵심.

---

## 학습 원칙

1. **진도보다 이해.** 한 단계 끝낼 때 "왜 이렇게 하지?"에 답할 수 있을 때 다음으로.
2. **단계 단위 커밋.** `git log`가 학습 일지가 되도록. 메시지에 "왜"를 짧게.
3. **단계 끝마다 `NOTES.md` 갱신.** 처음 본 C 문법 / 깨달은 CS 개념 / 삽질 기록.
4. **CS 기초는 병행 독서로 보완.** 아래 "추천 자료" 참고.
5. **24시간 룰.** 24시간 이상 막히면 검색·공식 문서·커뮤니티 질문. 혼자 붙잡지 말 것.

## Phase 1 — C와 친해지기 (목표 4~6주)

> "내 손으로 무언가 동작한다"의 즐거움을 먼저. 메모리·포인터의 첫 만남.

### 01. Snake (터미널 게임) ⭐ 진행 중

**만들 것**
- 터미널에서 화살표키로 조작하는 스네이크 게임
- 사과 먹으면 길어지고, 벽·자기 몸 부딪히면 게임오버

**만나는 CS 개념**
- 컴파일 → 링크 → 실행의 흐름
- 2D 배열의 메모리 레이아웃 (행 우선)
- 표준 입출력 / 버퍼링 / ANSI escape 코드
- 게임 루프 / 틱(tick) / `Sleep` vs busy-wait
- 구조체와 동적 메모리(`malloc`/`free`)
- 큐(FIFO) 자료구조
- 상태 머신 / 난수
- (선택) 연결 리스트 — 포인터의 진수

**Stage 분할** (각 단계 = 1 커밋)

| Stage | 내용 | 등장 CS 개념 |
|---|---|---|
| 0 | Hello World 컴파일 | 컴파일 → 링크 → 실행 |
| 1 | 정적 보드 그리기 | 2D 배열, 메모리 레이아웃 |
| 2 | 키 1회 입력 → 한 칸 이동 | 표준 입출력, 버퍼링, ANSI escape |
| 3 | 게임 루프 (자동 전진) | 무한 루프, 틱, Sleep |
| 4 | 사과 먹고 길어지기 | 구조체, malloc/free, 큐 |
| 5 | 충돌·게임오버·점수 | 상태 머신, 난수 |
| 6 (선택) | 배열 → 연결 리스트 리팩터 | 포인터, 연결 리스트 |

**체크 질문** (단계 끝마다)
- "`.c` → `.exe` 사이에 어떤 단계를 거치지?"
- "`board[3][5]`의 메모리 주소는 어떻게 계산되지?"
- "`getchar()`는 왜 Enter를 눌러야 반응하지?"
- "`malloc`한 메모리를 `free` 안 하면 무슨 일이?"

---

### 02. RPN 계산기 (역폴란드 표기 계산기)

**만들 것**
- `3 4 + 5 *` 같은 후위 표기 수식을 계산하는 CLI 프로그램
- 확장: 중위 표기(`(3 + 4) * 5`)를 후위로 변환 (Shunting-yard 알고리즘)

**만나는 CS 개념**
- **스택** 자료구조 (배열 기반 / 연결 리스트 기반 둘 다)
- 토큰화(파싱)의 첫 단계
- 후위 표기법이 컴파일러에서 왜 쓰이는지
- 함수 호출 스택과의 연결

**체크 질문**
- "왜 후위 표기가 계산기에 편할까?"
- "스택이 가득 차면(stack overflow) 무슨 일이?"

## Phase 2 — 시스템과 도구 (목표 4~6주)

> 운영체제와 한 발짝 가까워진다. 파일·프로세스·바이트.

### 03. Mini wc / cat / grep (유닉스 도구 클론)

**만들 것**
- `wc`: 파일의 줄 수·단어 수·바이트 수 세기
- `cat`: 파일 내용 출력
- `grep`: 파일에서 특정 문자열이 들어간 줄만 출력

**만나는 CS 개념**
- 파일 IO (`fopen`/`fread`/`fclose`)
- 시스템 콜이 뭔지, 표준 라이브러리와의 차이
- 표준 입출력 / 파이프 (`./mygrep "foo" < input.txt`)
- 문자 인코딩 (ASCII vs UTF-8) 첫 만남

**체크 질문**
- "`fread`가 한 번에 큰 파일 전체를 안 읽고 버퍼 단위로 읽는 이유는?"
- "표준 입력/출력은 어디서 오고 어디로 가지?"

---

### 04. 간단한 압축기 (RLE → 허프만)

**만들 것**
- 1단계: Run-Length Encoding (`AAABBC` → `A3B2C1`)
- 2단계: 허프만 코딩으로 진짜 압축

**만나는 CS 개념**
- **비트 연산** (`&`, `|`, `^`, `<<`, `>>`)
- 바이트와 비트의 차이를 코드로 체감
- **트리** 자료구조, 우선순위 큐
- 알고리즘이란 무엇인가 (시간 복잡도 첫 등장)
- 파일 포맷 설계 (헤더 + 페이로드)

**체크 질문**
- "왜 모든 데이터가 압축되지는 않을까?"
- "허프만 트리는 왜 빈도가 높은 글자에 짧은 코드를 줄까?"

## Phase 3 — 컴퓨터의 내부 (목표 6~10주)

> "컴퓨터가 어떻게 돌아가는가"를 직접 시뮬레이션해본다.

### 05. CHIP-8 에뮬레이터

**만들 것**
- 70년대 8비트 가상 게임기 CHIP-8을 C로 시뮬레이션
- 35개 명령어 구현 → Tetris·Pong 같은 옛날 ROM이 실제로 돌아감

**만나는 CS 개념**
- **CPU 사이클**: fetch → decode → execute
- 레지스터·메모리·스택·프로그램 카운터의 의미
- 명령어 집합(ISA)이 무엇인지
- 비트 연산 활용 (opcode 디코딩)
- 어셈블리어와의 연결고리

**체크 질문**
- "CPU의 레지스터는 왜 있는 걸까? 메모리만 있으면 안 되나?"
- "fetch-decode-execute의 각 단계는 실제 CPU에서 얼마나 걸릴까?"

---

### 06. (확장 후보) Mini Shell 또는 Bytecode VM

> Phase 3까지 끝나는 시점에서 흥미 가는 쪽으로 다시 결정.

- **Mini Shell**: 프로세스 / `fork`·`exec` / 파이프 / 시그널 — 운영체제의 정수 (단, WSL/Linux 환경 필요)
- **Bytecode VM**: 자체 명령어를 정의하고 실행하는 가상머신 — 컴파일러·인터프리터 입문
